# 💉 Датавак 2.0 (Work in Progress — MVP)

<img src="https://github.com/Aliam-Tukhvatulin/DataVac/blob/96d92ed9b9689cfc28dec2a5bbabfc8696d68767/logo.png" alt="Датавак 2.0 Logo" width="250" height="250">

Датавак 2.0 — веб-каталог вакцин, зарегистрированных в РФ. Проект по замене текущей версии (Битрикс) на современный стек (Python/Django + PostgreSQL).

## Ситуация
АНО «Коллективный иммунитет» с 2020 года развивает проект «Датавак» — специализированный справочник вакцин для медработников, студентов и пациентов. Текущая версия на Битриксе с Google-таблицами в качестве базы данных исчерпала возможности масштабирования: невозможно добавить модерацию контента, версионирование, автоматическое обновление инструкций. Перед командой стояла задача спроектировать новую версию системы с полноценной архитектурой, ролевой моделью и API.

## Задача
Необходимо было провести полный цикл анализа требований: от интервью с заказчиком и разбора текущей БД до спецификации требований, проектирования модели данных и API. Результат — пакет аналитических артефактов, готовый к передаче команде разработки.

## Решение
Проведён анализ текущей системы (Битрикс, Google-таблицы), прототипов в Figma, брифа заказчика и таблицы вопросов-ответов. На основе анализа выделен MVP. В MVP вошли: универсальный поиск с автоподстановкой, просмотр карточек вакцин, фильтрация и сортировка списков, административная панель с модерацией контента, управление популярностью поисковых подсказок, парсинг данных из ОХЛП/ГРЛС. Разработаны следующие артефакты системного анализа:

- Use Cases (32 сценария) — детальное описание поведения системы для всех ролей.
- Спецификация требований к ПО (SRS) — функциональные требования (100+), нефункциональные (46), бизнес-правила (8), глоссарий, описание логики фронта и бэка.
- ER-диаграмма (20+ сущностей, словарь данных) — спроектирована с учётом модерации через версии (vaccine_versions) и изоляции сервиса парсинга.
- Спецификация REST API (OpenAPI 3.0, 50+ эндпоинтов) — публичные и административные эндпоинты с полным описанием запросов, ответов и кодов ошибок.
- Sequence Diagrams (публичные и административные сценарии) — UML-диаграммы последовательности.
- Карта документации и журнал решений (20+ допущений с обоснованием).

## Использованные инструменты и нотации
- **Анализ требований:** интервью с заказчиком, разбор текущей БД и прототипов, BRD, SRS
- **Моделирование данных:** ERD (Crow’s Foot), словарь данных
- **Проектирование API:** REST, OpenAPI 3.0 (Swagger), JWT-аутентификация
- **Моделирование процессов:** UML (Use Case, Sequence Diagram), PlantUML
- **Документирование:** Google Docs, draw.io, VS Code (YAML)
- **Прочее:** Git, GitHub
